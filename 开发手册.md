## 1. 项目概述

WeChat EXP 是一款 Windows 平台下的微信 4.x 聊天记录提取、解密与分析工具。核心功能包括：

- **内存密钥提取**：从微信进程内存中提取 SQLCipher 4 数据库密钥
- **批量解密**：AES-256-CBC 解密微信消息数据库
- **Web 聊天查看器**：基于 Flask 的 SPA 聊天记录浏览器，支持头像、群组、消息搜索
- **V2 图片解密**：支持 WeChat 4.x 的 V2 格式 .dat 加密图片本地解码
- **年度报告**：8 张 Spotify-Wrapped 风格的数据卡片
- **多种导出**：词云、HTML 综合报告、员工报表、TXT/HTML 聊天导出

**技术栈**：Python 3.8+ | Flask 3.x | SQLite (SQLCipher 4) | Vanilla JS (SPA) | ECharts | pycryptodome

**兼容性**：Windows 10/11, WeChat 4.x（已测试 4.1.9 / 4.1.10，其他版本未经测试）

## 2. 系统架构

```
┌──────────────────────────────────────────────────────────────┐
│                        CLI (main.py)                          │
│   backup │ serve │ export │ harvest-keys                     │
├──────────────────────────────────────────────────────────────┤
│                    Web Application (web/)                     │
│   Flask SPA │ SSE Progress │ REST API │ Blueprints           │
├──────────────────────────────────────────────────────────────┤
│                    Engine Layer (engine/)                     │
│   decrypt │ services (chat/message/media/v2_key) │ parsers   │
├──────────────────────────────────────────────────────────────┤
│                   Backup Pipeline (backup/)                   │
│   scanner → decryptor → migrator → indexer → pipeline        │
├──────────────────────────────────────────────────────────────┤
│                   External Dependencies                       │
│   SQLCipher 4 │ zstandard │ pycryptodome │ Whisper │ ECharts  │
└──────────────────────────────────────────────────────────────┘
```

**分层设计原则**：

1. **Engine 层**是核心，被 CLI 和 Web 共享，不依赖 Flask
2. **Backup 管线**编排备份任务，依赖 Engine 层进行解密
3. **Web 层**仅做展示与路由，业务逻辑全部在 Engine 层
4. **CLI** 是最薄的入口层，解析参数后调用对应模块

## 4. 核心模块详解

### 4.1 引擎层 (engine/)

#### 4.1.1 decrypt.py — 数据库解密

```python
from engine.decrypt import decrypt_database

# SQLCipher 4 使用 AES-256-CBC，512,000 次 PBKDF2 迭代
# 密钥为 64 字符 hex 字符串（32 字节）
success = decrypt_database(src_path, dst_path, key_bytes)
```

微信 4.x 使用 SQLCipher 4 加密所有消息数据库。解密模块负责：
- 使用 64 位 hex 密钥进行 AES-256-CBC 解密
- 支持批量解密（message_*.db, contact.db, session.db 等）
- 自动密钥匹配（按文件名前缀 → 任意可用密钥的降级策略）

#### 4.1.2 services/chat.py — 联系人查询

```python
from engine.services.chat import get_contacts, get_group_info

# 获取所有联系人（包含头像、备注名、群组信息）
contacts = get_contacts(decrypted_dir, wxid)

# 获取群组详细信息（成员列表、群名等）
info = get_group_info(decrypted_dir, chat_id)
```

5 级名称解析优先级：备注名 → 群昵称 → 微信号 → 通讯录名 → 原始 wxid

#### 4.1.3 services/message/ — 消息查询

```python
from engine.services.message import query_messages, query_message_detail

# 分页查询消息（支持日期范围、类型、发送者、关键词过滤）
result = query_messages(
    decrypted_dir, chat_id, wxid=wxid,
    page=1, per_page=50,
    start_date='2026-05-01', end_date='2026-05-27',
    msg_types='3,43',  # 图片+视频
    keyword='搜索词'
)

# 获取单条消息详情（含所有字段）
detail = query_message_detail(decrypted_dir, msg_id, chat_id='xxx')
```

消息处理管线：
1. **zstd 解压**：`message/decode.py` — 检测 `\x28\xb5\x2f\xfd` 魔数并解压
2. **类型解析**：`parsers/types.py` — 用正则提取 XML 中的媒体信息
3. **名称解析**：`services/name_resolver.py` — 统一 wxid → 显示名
4. **媒体路径**：`message/media_resolve.py` — 多级路径解析

#### 4.1.4 services/media.py — 媒体提供服务

```python
from engine.services.media import (
    serve_hardlink_media,  # HardLink DB 路径解析 + 解密
    serve_media,           # 传统路径解析
    serve_voice,           # 语音 SILK → WAV
    transcribe_voice,      # Whisper 语音转文字
    _silk_to_wav,          # SILK V3 解码（调用 silk_decoder.exe）
)
```

**语音解码流程**（`serve_voice` → `_silk_to_wav`）：
1. 搜索缓存目录 `{decrypted_dir}/media/voice/`
2. 未找到则从 `VoiceInfo` 表提取（`_extract_voice_from_db`）
3. 调用 `silk_decoder.exe` 将 SILK V3 → PCM
4. `_write_wav()` 将 PCM 封装为 WAV（24000Hz, 16-bit, mono）
5. 失败时输出 `[WARN]` 日志（含具体原因），fallback 返回 raw SILK

**调试语音播放问题**：查看 Flask 控制台的 `[WARN]` 输出，常见原因：
- `silk_decoder.exe not found` → 源路径检查 `tools/`，PyInstaller 检查 `_MEIPASS/tools/`
- `silk_decoder.exe failed (exit=X)` → SILK 文件头无效或损坏
- 无 WARN 且返回 206 → 文件已找到但转换失败（可能是子进程权限问题）

**PyInstaller 打包注意**：`silk_decoder.exe` 需在 `build.bat` 中通过 `--add-data` 和 `.spec` 文件中通过 `datas` 列表纳入打包。

支持三种 .dat 加密版本：

| 版本 | 签名 | 加密方式 | 解密能力 |
|------|------|----------|----------|
| V0 | 无签名 | 纯 XOR | 自动检测密钥（0xC9 等） |
| V1 | `\x07\x08V1\x08\x07` | AES-128-ECB (固定) + XOR | 内置密钥，可解密 |
| V2 | `\x07\x08V2\x08\x07` | AES-128-ECB (动态) + XOR | 需内存收割 |

### 4.2 备份管线 (backup/)

#### 管线编排


```python

    start_date=None,    # 起始日期 "YYYY-MM-DD"

    on_progress=None,   # 进度回调 fn(stage, detail, progress)

# → {'success': bool, 'stats': {...}, 'errors': [...]}
```

### 4.3 Web 应用 (web/)

#### 导出 API 智能匹配 (`routes/export_api.py`)

导出和词云的 `/chat` 和 `/wordcloud` 端点都支持联系人智能匹配：

1. **精确匹配**：username 或 display_name 相等 → 直接使用
2. **模糊匹配**：在 display_name 和 username 中搜索子串
   - 单个匹配 → 自动选中
   - 多个匹配 → 通过 SSE `push.select(matches)` 发送匹配列表给前端
   - 零匹配 → `push.error()` 返回错误
3. **前端选择**：用户从匹配列表中点击选择后，WizardPage 将 username 回填到输入字段，自动重发请求（此时精确匹配命中）

`push.select()` 发送的 SSE 事件被 `sse_progress.js` 解析，触发 WizardPage 的 `onSelect` → `handleSelect()` 回调。

## 5. 数据流与关键流程

### 5.1 备份流程

```
1. scanner.py
   扫描 db_storage/ 目录
   ├── 发现所有微信账户（wxid_xxx）
   ├── 枚举每个账户的数据库（message_0.db, contact.db...）
   └── 识别会话列表

2. decryptor.py
   对每个加密数据库：
   ├── 查找对应密钥（config → 自动扫描）
   ├── SQLCipher 4 AES-256-CBC 解密
   └── 输出到 output/message/*.db

3. migrator.py
   扫描解密后的消息数据库：
   ├── 查询所有媒体引用（图片/视频/文件/语音）
   ├── 通过 HardLink DB 解析路径
   ├── 硬链接（同分区）或复制（跨分区）到 output/media/
   └── 支持日期范围过滤

4. indexer.py
   构建统一索引：
   ├── 扫描所有 message_*.db 中的 Msg_* 表
   ├── 生成聚合统计（消息数、日期范围、类型分布）
   └── 写入 output/chats.db
```

### 5.2 消息查询流程

```
GET /api/messages?chat_id=xxx&page=1&per_page=50

1. Flask route → engine/services/message/__init__.py
2. 确定消息表：根据 chat_id 找到对应 Msg_* 表
3. SQL 查询：
   ├── WHERE 子句组合（日期、类型、发送者、关键词）
   ├── ORDER BY create_time DESC
   └── LIMIT/OFFSET 分页
4. 每条消息：
   ├── message/decode.py        → zstd 解压
   ├── parsers/types.py         → XML 解析（提取媒体信息）
   ├── services/name_resolver.py → wxid → 显示名
   └── message/media_resolve.py → 媒体路径（用于 API URL 构建）
5. 返回 JSON：{messages: [...], pagination: {...}}
```

### 5.3 媒体解析流程

```
GET /api/hardlink-media?md5=xxx&path=xxx&type=3

1. _resolve_hardlink_path()
   ├── 使用提供的 local_path（如果有）
   ├── HardLink DB 查询（md5 → dir1/dir2 → 路径）
   └── 多存储根目录尝试（D:\xwechat_files, C:\xwechat_files）

2. 版本检测
   ├── 读取文件头 6 字节
   ├── V2 签名 → 走 V2 解密管线
   ├── V1 签名 → AES-128-ECB（固定密钥 cfcd208495d565ef）+ XOR
   └── V0（无签名） → XOR 自动检测密钥

3. V2 解密尝试顺序
   ├── ① 缓存键映射（_media_keys.json）
   ├── ② 消息 DB 逐个搜索（_collect_image_aeskey）
   ├── ③ 微信进程内存提取（find_keys_for_files）
   └── ④ 缩略图缓存降级（本地缓存目录）

4. 成功后缓存到 decrypted_media/ 子目录
```

### 5.4 V2 密钥收割流程

```
python main.py harvest-keys

1. _load_v2_ciphertexts()
   ├── 扫描 output/media/images/*.dat
   ├── 读取每个 V2 文件的 AES 密文块
   └── 构建 {md5: (file_path, ciphertext_16b)}

2. 收割循环（每 N 秒）
   ├── 检查微信进程是否存在
   ├── 策略 1：扫描 V2 头部附近内存
   │   ├── 找到所有 \x07\x08V2\x08\x07 出现位置
   │   ├── 读取 ±256 字节窗口
   │   └── 16 字节滑动窗口测试 → 所有密文
   ├── 策略 2：扫描所有 RW 内存
   │   ├── 正则匹配 32 字符 hex 字符串
   │   └── hex 解码 → 测试所有密文
   └── 找到的密钥 → 立即缓存到 _media_keys.json

3. 自动停止条件
   ├── 所有 V2 密钥已找到
   ├── 微信退出
   └── 60 轮无新密钥
```

## 6. V2 图片解密系统

### 6.1 V2 文件格式

```
偏移    大小    内容
0       6       签名 \x07\x08\x56\x32\x08\x07
6       4       AES 数据大小（LE uint32）
10      4       XOR 数据大小（LE uint32）
14      1       填充字节
15      N       AES-128-ECB 加密数据（PKCS7 填充）
15+N    16      常量尾部（56fbf486095aa2eed54a41405dffe35d）
15+N+16 M       XOR 加密尾部（密钥 0xC9）
```

### 6.2 密钥来源

| 来源 | 可靠性 | 说明 |
|------|--------|------|
| 消息 XML `aeskey` 属性 | 0%（已验证） | CDN aeskey ≠ 本地文件密钥 |
| 微信进程内存 | 100%（浏览时） | 仅在图片查看期间存在 |
| `_media_keys.json` 缓存 | 100%（收割后） | 收割器缓存结果 |

### 6.3 收割器三种策略

**策略 1：V2 头部邻近扫描**
- 搜索所有内存中的 V2 魔数
- 读取每个出现位置 ±256 字节
- 16 字节滑动窗口测试原始二进制密钥
- 32 字节 hex 解码窗口测试

**策略 2：wx_key 模式定位**
- 检测微信版本号
- 在 Weixin.dll 中搜索版本特定的字节模式
- 定位图片解密函数地址
- 读取函数附近内存寻找密钥

**策略 3：全内存正则扫描**
- 枚举所有 RW 内存区域
- 正则匹配 32 字符 hex 字符串
- hex 解码后测试 AES 解密

## 附录 C：微信 4.x 数据库文件与表结构

微信 4.x 将用户数据存储在 `xwechat_files/<wxid>/db_storage/` 目录下，所有数据库使用 **SQLCipher 4**（AES-256-CBC）加密。本工具在备份流程中解密以下数据库并重新组织到 `output/decrypted/` 目录。

### C.1 数据库文件清单

| # | 数据库文件 | 微信原始路径 | 解密后路径 | 用途 |
|---|-----------|-------------|-----------|------|
| 1 | `message_0.db` ~ `message_N.db` | `db_storage/message/` | `decrypted/message/` | 消息分片存储（核心） |
| 2 | `contact.db` | `db_storage/contact/` | `decrypted/contact/` | 联系人资料与头像 |
| 3 | `session.db` | `db_storage/session/` | —（直接读取） | 会话摘要与最近消息预览 |
| 4 | `hardlink.db` | `db_storage/hardlink/` | `decrypted/hardlink/` | 媒体文件 HardLink 路径映射 |
| 5 | `media_0.db` | `db_storage/message/` | `decrypted/message/` | 语音消息原始数据 |
| 6 | `message_resource.db` | `db_storage/message/` | `decrypted/message/` | 消息媒体文件引用索引 |
| 7 | `chats.db` | —（本工具生成） | `decrypted/data/` | 统一聊天索引（备份产物） |

> **注**：`session.db` 可能未加密，工具直接从原始路径或解密目录读取。备份管线当前不解密 `session.db`。

### C.2 message_N.db — 消息分片数据库

微信按时间/容量将消息拆分到多个分片文件（`message_0.db`, `message_1.db`, ...）。每个分片包含多个 `Msg_<hash>` 表，每个表对应一个聊天会话。

#### 表：`Msg_<md5hash>`

表名 = `Msg_` + `md5(chat_id)` 的 32 位十六进制哈希。

| 列名 | 类型 | 说明 |
|------|------|------|
| `local_id` | INTEGER | 消息在聊天内的本地 ID（主键） |
| `local_type` | INTEGER | 消息类型码（低 16 位为实际类型，见附录 A） |
| `origin_source` | INTEGER | 1 = 自己发送（群聊可靠，单聊不可靠） |
| `create_time` | INTEGER | Unix 时间戳（秒；部分旧数据为毫秒 > 1000000000000） |
| `status` | INTEGER | 消息状态（已发送/已读等） |
| `message_content` | BLOB/TEXT | 消息内容：zstd 压缩 XML（魔数 `\x28\xb5\x2f\xfd`）、纯文本（UTF-8/GBK）、或 protobuf 二进制 |
| `real_sender_id` | INTEGER | 发送者在聊天内的成员索引（**不是** contact.db 的 rowid） |
| `packed_info_data` | BLOB | protobuf 编码的媒体元数据（图片/视频/文件的 md5、尺寸等） |
| `compress_content` | BLOB | 备用压缩内容列（部分消息使用） |
| `server_id` | INTEGER | 服务端消息 ID |
| `sort_seq` | INTEGER | 排序序列号 |

**zstd 压缩说明**：微信 4.x 对类型 3/6/42/43/47/48/49/50 的消息内容使用 zstd 压缩。压缩数据以 `\x28\xb5\x2f\xfd`（zstd 帧头）开头，解压后得到 XML 文本。单聊中接收的消息在 XML 前有 `wxid_xxx:\n` 发送者前缀。

#### 表：`Name2Id`

| 列名 | 说明 |
|------|------|
| `user_name` | 微信用户名（wxid 或微信号） |

用于将 `Msg_<hash>` 表的哈希值反向映射回 `chat_id`：遍历所有 `user_name`，计算 `md5(user_name)` 匹配哈希。

#### 表：`ChatName2Id`

| 列名 | 说明 |
|------|------|
| `rowid` | 整数 ID |
| `user_name` | 微信用户名 |

将 `chat_id` 映射为整数，供 `MessageResourceInfo` 表引用。

| `create_time` | 消息创建时间戳 |

| `voice_data` | SILK V3 格式音频原始数据（BLOB） |

语音播放流程：从 `VoiceInfo` 提取 `voice_data` → 缓存为 `.silk` 文件 → `silk_decoder.exe` 解码为 PCM → 封装 WAV 返回前端。

### C.7 message_resource.db — 消息资源索引

#### 表：`MessageResourceInfo`

| 列名 | 说明 |
|------|------|
| `chat_id` | 聊天 ID（引用 `ChatName2Id.rowid`） |
| `message_svr_id` | 服务端消息 ID |
| `message_local_id` | 本地消息 ID |
| `message_local_type` | 消息类型 |
| `message_create_time` | 消息创建时间 |
| `packed_info` | protobuf 编码的文件引用（`{'2': {'1': b'file_name_hash'}}`） |

当 `packed_info_data` 中无 md5 或 HardLink DB 查不到时，通过此表获取 `file_name` 哈希，再在 HardLink DB 中模糊匹配文件路径。

#### 表：`ChatName2Id`

同 message_N.db 中的 `ChatName2Id`，用于 `chat_id` → 整数 ID 映射。

### C.8 chats.db — 统一聊天索引（本工具生成）

备份管线第四阶段（indexer）生成，位于 `decrypted/data/chats.db`。

```sql
CREATE TABLE chats (
    chat_id TEXT PRIMARY KEY,        -- 聊天 ID
    display_name TEXT,               -- 预解析的显示名
    message_count INTEGER DEFAULT 0, -- 消息总数
    first_msg_time INTEGER,          -- 最早消息时间戳
    last_msg_time INTEGER,           -- 最新消息时间戳
    is_group INTEGER DEFAULT 0,      -- 是否群聊
    source_db TEXT                   -- 来源 message_N.db 文件名
);

CREATE TABLE messages (
    chat_id TEXT,                    -- 聊天 ID
    local_id INTEGER,                -- 消息本地 ID
    local_type INTEGER,              -- 消息类型
    create_time INTEGER,             -- 消息时间戳
    PRIMARY KEY (chat_id, local_id)
);
```

Web 查看器优先从 `chats.db` 读取联系人列表，缺失时才回退到全量扫描 `message_N.db`。

### C.9 消息内容编码说明

微信 4.x 的 `message_content` 列存在多种编码格式：

| 格式 | 判定方式 | 解码 |
|------|---------|------|
| zstd 压缩 XML | 前 4 字节 = `\x28\xb5\x2f\xfd` | zstd 解压 → UTF-8/GBK |
| 纯文本 UTF-8 | 字节内容可解码为 UTF-8 | 直接 decode |
| 纯文本 GBK | UTF-8 解码失败 | GBK decode（含 CDATA 修复） |
| protobuf 二进制 | UTF-8 解码后大量 `�` 替换字符 | 使用 `blackboxprotobuf` 解码 |
| protobuf + XML 混合 | 类型 10000/10002/49 | 从二进制中提取 XML 片段 |

### C.10 单聊发送者识别机制

微信 4.x 的单聊消息不直接标注发送者，需通过以下策略推断：

1. **发送者前缀扫描**（Strategy 1 & 2）：文本消息中 `wxid_xxx:\n` 前缀标识对方发送
2. **origin_source 统计**（Strategy 3）：`origin_source=1` 最多的 `real_sender_id` 为本人
3. **zstd XML 中的 fromusername**（Strategy 4）：媒体消息（类型 47/49）的 XML 中含发送者信息
4. **逐消息启发式**（Fallback）：纯文本无前缀 → 自发送；有前缀 → 对方发送

> **注意**：`real_sender_id` 是**每个消息分片内**的成员索引，不同 `message_N.db` 之间
> 同一联系人的 rsid 值可能不同，必须为每个分片独立构建 sender_map。
> 旧版 `chat_export.py` 曾使用 `Name2Id` 表的 rowid 进行 wxid 查找，但该表存储的
> wxid 不含账号后缀（如 `wxid_xxx` vs `wxid_xxx_10e8`），导致字符串完全匹配失败。
> 现已统一使用 `_build_sender_map()` 函数。

群聊消息在内容前统一携带 `发送者wxid:\n` 前缀，识别相对简单。
